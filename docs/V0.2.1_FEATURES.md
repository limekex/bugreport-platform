# Version 0.2.1 - New Features

## Overview

Two major UX improvements added to the bug report widget based on user feedback:

1. **Screenshot Region Selection** - Capture specific areas instead of just fullscreen
2. **Console Errors Display** - Show users what console errors are being captured

---

## ✅ Feature 1: Screenshot Region Selection

### What's New

Users can now choose between two screenshot modes:

- **📸 Full screen** - Captures the entire page (previous behavior)
- **✂️ Select region** - Drag to select a specific screen area

### How It Works

1. User clicks "Select region" button
2. A semi-transparent overlay appears with crosshair cursor
3. User clicks and drags to select the desired area
4. A blue selection box visualizes the selection
5. On release, the selected region is captured
6. Annotation editor opens with just the selected region

### User Controls

- **ESC key**: Cancel selection and return to modal
- **Minimum size**: 20x20 pixels (smaller selections are rejected)
- **Visual feedback**: Instructions shown at top, selection box updates in real-time

### Technical Implementation

**New File: `packages/widget-sdk/src/utils/regionSelector.ts`**

```typescript
// Main functions
export async function selectScreenRegion(): Promise<RegionSelection | null>
export async function captureRegion(region: RegionSelection): Promise<string | null>

// Types
export type ScreenshotMode = 'fullscreen' | 'region';
export interface RegionSelection {
  x: number;
  y: number;
  width: number;
  height: number;
}
```

**Updated: `packages/widget-sdk/src/widget/modal.ts`**
- Split single "Capture screen" button into two buttons
- Enhanced `handleCaptureScreenshot()` with mode parameter
- Added CSS for secondary button style

### Benefits

- **More precise bug reports**: Users can highlight specific UI elements
- **Smaller file sizes**: Region captures are smaller than fullscreen
- **Better privacy**: Users can exclude sensitive information from other parts of the screen
- **Professional workflow**: Similar to tools like Snagit, Greenshot, etc.

## ✅ Feature 2: Console Errors Display

### What's New

The modal now displays captured console errors **before** submission, so users can see what's being sent.

### What's Shown

When console errors are present:
```
Recent console errors (3 captured)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
WILL BE INCLUDED IN BUG REPORT:
10:34:22 AM: TypeError: Cannot read property 'name' of undefined
10:34:25 AM: ReferenceError: myFunction is not defined
10:35:01 AM: Error: Network request failed
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

When no errors:
```
Recent console errors
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
No recent console.error() calls detected
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
We automatically capture console.error() calls to help diagnose issues
```

### Important Notes

**Only `console.error()` is captured** - not `console.log()`, `console.warn()`, etc.

This is intentional because:
- `console.log()` is often used for debugging/development noise
- `console.error()` represents actual problems
- Focusing on errors keeps reports actionable

### Display Behavior

- Shows up to **5 most recent errors** in the modal
- All captured errors (up to 20) are still sent to GitHub
- Each error shows: timestamp + message (truncated to 100 chars for UI)
- Full message and stack trace included in GitHub issue

### Technical Implementation

**New Function in `modal.ts`:**

```typescript
function buildConsoleErrorsSection(): string {
  const errorsJson = getCollectedErrors();
  // Parses JSON, formats HTML section
  // Shows max 5 errors with timestamps
  // Escapes HTML to prevent XSS
}
```

**Styling:**
- Red border and pink background (`#fef2f2` with `#fecaca` border)
- Monospace font for error messages
- Scrollable if many errors
- Clear visual hierarchy

### Benefits

- **Transparency**: Users know exactly what diagnostic data is being sent
- **Verification**: Users can confirm errors are related to their bug
- **Education**: Users learn which errors are being tracked
- **Trust**: No hidden data collection

## FAQ

### Q: Why only console.error() and not console.log()?

**A:** `console.log()` is typically used for development debugging and would add too much noise. `console.error()` represents actual application errors that are relevant to bug reports.

If you need to capture custom debug info, use `console.error()` for important issues.

### Q: Can users disable console error capture?

**A:** Not currently. The console errors help developers diagnose issues. However, users can see exactly what's being sent before they click "Submit".

Future enhancement: Add a checkbox to opt-out of console error capture.

### Q: What if the region selector overlay doesn't appear?

**A:** This could happen if:
- JavaScript errors prevented loading
- CSS conflicts with your site
- Z-index issues with other overlays

The widget uses a very high z-index (999998) to stay on top. Check browser console for errors.

### Q: Does region selection work on mobile?

**A:** The overlay is touch-enabled and should work on mobile, but the UX may be less precise. Mobile users can still use fullscreen capture or file upload.
